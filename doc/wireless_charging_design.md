# 无线充电系统通讯技术方案

> 日期：2026-05-05
> 状态：设计阶段

## 1. 系统概述

### 1.1 系统组成

**一发四收**无线充电系统：
- 1 个发射端（发送端功率板 + 基站端通讯板）
- 4 个接收端（接收端功率板 + 设备端通讯板）

| 组件 | 数量 | 角色 | 说明 |
|------|------|------|------|
| 发送端功率板 | 1 | 充电发射 | 持续发送 CAN 报文到基站端 |
| 基站端通讯板 | 1 | 主控/网关 | 缓存 CAN 报文，接收设备端红外数据 |
| 接收端功率板 | 4 | 充电接收 | 持续发送 CAN 报文到设备端 |
| 设备端通讯板 | 4 | 终端/中继 | 缓存 CAN 报文，通过红外发送到基站端 |

### 1.2 通讯链路

```
发送端功率板 ──CAN(持续)──→ 基站端通讯板 ←──IR(单向)── 设备端通讯板 ←──CAN(持续)── 接收端功率板
     ↑                          │                          │                      │
     │                          │                          │                      │
     └──────────────────────────┘                          └──────────────────────┘
         CAN 应答                                              CAN 应答
```

### 1.3 核心原则

1. **CAN 缓冲**：通讯板持续接收 CAN 报文，缓存最新一帧
2. **红外单向**：设备端通过红外发送最新 CAN 报文到基站端
3. **设备轮询**：多个设备端轮流发送，避免冲突
4. **无握手**：设备端持续发送，基站端持续接收

### 1.4 数据流

```
┌─────────────────────────────────────────────────────────────────────────┐
│                        数据流向（单向红外）                                │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  上行链路（设备 → 基站）：                                               │
│  接收端功率板 ──CAN──→ [设备CAN缓冲区] ──IR──→ [基站CAN缓冲区] ──CAN──→ 发送端功率板
│                       (最新CAN帧)             (最新CAN帧)               │
│                                                                         │
│  基站端不需要通过红外发送数据到设备端                                     │
│  设备端通过红外单向发送自己的CAN数据                                      │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

## 2. 通讯方式设计

### 2.1 基础功能：红外透传 CAN 数据

**核心逻辑**：
- 基站持续接收发送端功率板的 CAN 报文，缓存最新一帧
- 设备持续接收接收端功率板的 CAN 报文，缓存最新一帧
- 基站通过红外轮询设备，发送最新 CAN 帧，请求设备回复
- 设备收到请求后，立即回复最新 CAN 帧

```
┌─────────────────────────────────────────────────────────────────────────┐
│                        红外透传 CAN 流程                                 │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  基站端                                    设备端 (目标: 0xD0)            │
│    │                                         │                          │
│    │  持续接收发送端功率板CAN报文              │  持续接收接收端功率板CAN报文  │
│    │  缓冲区: [CAN_ID=0x123, DATA=最新]      │  缓冲区: [CAN_ID=0x456, DATA=最新]
│    │                                         │                          │
│    │  50ms定时器触发                          │                          │
│    │                                         │                          │
│    │── IR: CAN_REQ ─────────────────────────→│                          │
│    │   [CAN_ID=0x123, DATA=基站最新数据]      │                          │
│    │                                         │                          │
│    │                                         │  收到请求，立即回复        │
│    │                                         │  读取缓冲区最新CAN帧       │
│    │                                         │                          │
│    │←── IR: CAN_RSP ─────────────────────────│                          │
│    │   [CAN_ID=0x456, DATA=设备最新数据]      │                          │
│    │                                         │                          │
│    │  收到回复，转发到发送端功率板              │                          │
│    │  更新缓冲区                              │                          │
│    │                                         │                          │
│    │  50ms后切换到下一个设备 (0xD1)           │                          │
│    │                                         │                          │
└─────────────────────────────────────────────────────────────────────────┘
```

### 2.2 CAN 缓冲区设计

**设计要点**：
- CAN 报文输入速率：1kHz（1ms 一帧）
- 红外轮询速率：20Hz（50ms 一周期）
- 50ms 内会收到约 50 帧 CAN 数据
- 缓冲区保存最近数据，红外传输时只取最新一帧

```c
// CAN 环形缓冲区
#define CAN_BUF_SIZE  64  // 保存最近 64 帧

typedef struct {
    bsp_twai_msg_t frames[CAN_BUF_SIZE];  // 环形缓冲区
    uint32_t head;                         // 写入位置
    uint32_t count;                        // 有效帧数
    SemaphoreHandle_t mutex;               // 互斥锁
} can_ring_buf_t;

// 基站端：缓存发送端功率板的CAN帧
static can_ring_buf_t s_tx_power_buf;  // 下行：基站→设备

// 设备端：缓存接收端功率板的CAN帧
static can_ring_buf_t s_rx_power_buf;  // 上行：设备→基站

// 写入新帧（CAN 接收回调中调用）
void can_buf_push(can_ring_buf_t *buf, const bsp_twai_msg_t *frame) {
    xSemaphoreTake(buf->mutex, portMAX_DELAY);
    buf->frames[buf->head] = *frame;
    buf->head = (buf->head + 1) % CAN_BUF_SIZE;
    if (buf->count < CAN_BUF_SIZE) buf->count++;
    xSemaphoreGive(buf->mutex);
}

// 读取最新帧（红外传输时调用）
bool can_buf_get_latest(can_ring_buf_t *buf, bsp_twai_msg_t *frame) {
    xSemaphoreTake(buf->mutex, portMAX_DELAY);
    if (buf->count == 0) {
        xSemaphoreGive(buf->mutex);
        return false;
    }
    uint32_t latest = (buf->head + CAN_BUF_SIZE - 1) % CAN_BUF_SIZE;
    *frame = buf->frames[latest];
    xSemaphoreGive(buf->mutex);
    return true;
}
```

### 2.3 CAN 过滤器配置

```c
// 配置 CAN 过滤器，只接收需要的 CAN ID
void can_filter_config(void) {
    twai_filter_config_t filter = {
        .acceptance_code = 0x123 << 21,  // 例：只接收 CAN ID 0x123
        .acceptance_mask = 0x7FF << 21,  // 11 位标准 ID 掩码
        .single_filter = true,
    };
    bsp_twai_set_filter(&filter);
}

// 或者接收所有帧（调试用）
void can_filter_accept_all(void) {
    twai_filter_config_t filter = {
        .acceptance_code = 0x00000000,
        .acceptance_mask = 0xFFFFFFFF,
        .single_filter = true,
    };
    bsp_twai_set_filter(&filter);
}
```

**过滤器配置建议**：
- 根据实际 CAN 协议配置过滤器
- 只接收需要的 CAN ID，减少 CPU 负载
- 可配置多个过滤器接收不同 ID 范围

### 2.3 帧格式

**红外 CAN 透传帧**：
```
┌──────────┬──────┬──────────┬──────────┬─────────────┬──────┬──────┐
│ header   │ ctrl │ master_id│ slave_id │ CAN 帧数据  │ seq  │ crc  │
│ 2 bytes  │ 1B   │ 1B       │ 1B       │ 13 bytes    │ 1B   │ 2B   │
│ 0xAA55   │ 0x18 │ 0xA0     │ 0xD0-0xD3│             │      │      │
└──────────┴──────┴──────────┴──────────┴─────────────┴──────┴──────┘
```

**CAN 数据封装**：
```
┌──────────┬──────────┬─────────────────────┐
│ CAN_ID   │ DLC      │ CAN_DATA            │
│ 4 bytes  │ 1 byte   │ 8 bytes (固定)       │
└──────────┴──────────┴─────────────────────┘
```

### 2.4 时序设计

**CAN 输入与红外轮询关系**：

```
时间轴: 0ms    10ms    20ms    30ms    40ms    50ms
         │       │       │       │       │       │
CAN输入: │▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓│  1kHz，每1ms一帧
         │       │       │       │       │       │
缓冲区:  │ [0] [1] [2] [3] [4] ... [48] [49]   │  50帧缓存
         │       │       │       │       │       │
红外:    │ REQ#0 │       │ REQ#1 │       │ REQ#2 │  20Hz，每50ms一轮
         │(最新) │       │(最新) │       │(最新) │  只取最新帧传输

关键：红外传输时只取缓冲区最新帧，不是传输所有缓存帧
```

**轮询时序**：

```
时间轴: 0ms    12.5ms  25ms    37.5ms  50ms    62.5ms  75ms    87.5ms  100ms
         │       │       │       │       │       │       │       │       │
基站:    │ REQ#0 │       │ REQ#1 │       │ REQ#2 │       │ REQ#3 │       │
         │   ↓   │ RSP#0 │   ↓   │ RSP#1 │   ↓   │ RSP#2 │   ↓   │ RSP#3 │
设备#0:  │ ───── │ ───── │       │       │       │       │       │       │
设备#1:  │       │       │ ───── │ ───── │       │       │       │       │
设备#2:  │       │       │       │       │ ───── │ ───── │       │       │
设备#3:  │       │       │       │       │       │       │ ───── │ ───── │

周期: 50ms (20Hz)
每设备占用: 12.5ms (请求 + 等待回复)
每轮请求: 取缓冲区最新帧
```

### 2.5 关键参数

| 参数 | 值 | 说明 |
|------|-----|------|
| CAN 输入速率 | 1kHz | 功率板持续发送，1ms 一帧 |
| CAN 缓冲区大小 | 64 帧 | 保存最近 64 帧（约 64ms 数据） |
| CAN 过滤器 | 可配置 | 只接收需要的 CAN ID |
| 红外轮询周期 | 50ms | 20Hz，每个设备每秒通讯 5 次 |
| 红外传输内容 | 最新帧 | 每次只取缓冲区最新一帧传输 |
| 单次超时 | 10ms | 等待设备回复超时 |
| 设备响应 | <5ms | 收到请求后立即回复 |
| CAN 速率 | 1Mbps | 与功率板通讯速率 |
| 红外速率 | 4800 baud | 红外链路速率 |

**数据丢失分析**：
- 50ms 内收到约 50 帧 CAN 数据
- 红外只传输最新 1 帧
- 49 帧在缓冲区中，可用于：
  - 重传（如果需要）
  - 历史记录
  - 统计分析

### 2.6 错误处理

| 情况 | 处理方式 |
|------|----------|
| 设备回复超时 | 跳过该设备，50ms 后重试 |
| CRC 错误 | 丢弃帧，等待下次轮询 |
| CAN 总线错误 | 自动恢复，继续使用缓冲区数据 |
| 缓冲区无数据 | 发送空帧或跳过 |

### 2.2 扩展功能：ESP-NOW 透传 CAN 数据

**适用场景**：长距离、高速率（后续扩展）
**核心特点**：通过红外识别确认目标设备后，建立 ESP-NOW 点对点连接透传 CAN 数据

```
┌─────────────────────────────────────────────────────────────────────────┐
│                        ESP-NOW 透传 CAN 流程（扩展）                      │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  基站端                                    设备端 (目标: 0xD0)            │
│    │                                         │                          │
│    │  1. 红外识别确认目标设备                  │                          │
│    │── IR: IDENTIFY_REQ ────────────────────→│                          │
│    │←── IR: IDENTIFY_RSP ────────────────────│  (携带 MAC 地址)          │
│    │                                         │                          │
│    │  2. 建立 ESP-NOW 连接                    │                          │
│    │═══ ESP-NOW: PAIR ═════════════════════→│                          │
│    │←══ ESP-NOW: PAIR_ACK ══════════════════│                          │
│    │                                         │                          │
│    │  3. 透传 CAN 数据                        │                          │
│    │═══ ESP-NOW: CAN_REQ [最新CAN帧] ══════→│                          │
│    │←══ ESP-NOW: CAN_RSP [最新CAN帧] ═══════│                          │
│    │   (延迟 <5ms, 可靠性 >99%)              │                          │
│    │                                         │                          │
│    │  4. 完成后可切换设备                     │                          │
│    │                                         │                          │
└─────────────────────────────────────────────────────────────────────────┘
```

**两种方式对比**：

| 特性 | 红外透传 (基础) | ESP-NOW 透传 (扩展) |
|------|-----------------|---------------------|
| 数据类型 | CAN 帧 | CAN 帧 |
| 传输距离 | ~10m | ~200m |
| 传输速率 | 4800 baud | 1Mbps+ |
| 延迟 | ~100ms | <5ms |
| 可靠性 | ~65% | >99% |
| 设备识别 | 红外广播 | 红外识别 + ESP-NOW 配对 |
| 实现优先级 | **高** | 低（后续扩展） |

## 5. 错误处理

### 5.1 CAN 总线错误

| 错误类型 | 处理方式 |
|----------|----------|
| Bus-off | 自动恢复，重新初始化驱动 |
| CRC 错误 | 丢弃帧，等待下一次轮询 |
| 超时 | 标记设备忙碌，跳过本次轮询 |

### 5.2 红外通讯错误

| 错误类型 | 处理方式 |
|----------|----------|
| CRC 错误 | 丢弃帧，等待下一次轮询 |
| 超时 | 标记设备离线，3 次超时后移除 |
| 回声干扰 | 发送后 50ms 静默窗口 |

### 5.3 ESP-NOW 错误

| 错误类型 | 处理方式 |
|----------|----------|
| 发送失败 | 重试 3 次，失败后断开连接 |
| 设备离线 | 心跳超时 6s 标记离线 |
| 信号弱 | 降低发送速率，增加重试 |

## 10. 测试记录

### 10.1 红外双向通讯测试（失败）

**测试时间**：2026-05-06

**测试方案**：
- 基站端通过红外发送 CMD 帧到设备端
- 设备端接收后回复 RSP 帧
- 使用 4800/1200 波特率

**测试结果**：❌ 失败

**失败现象**：
1. 设备端能正确接收基站的 CMD 帧
2. 设备端发送 RSP 帧后，收到自己发送的回声帧，但数据被损坏
3. 基站端无法接收到设备端的 RSP 帧

**日志示例**：
```
设备端：
I (xxx) ir_slave: send_rsp: to=0xa0 from=0xd0 len=13 ctrl=0x20 payload_len=20 crc=0xfe5f
I (xxx) ir_slave: send_rsp OK: written=22
W (xxx) app_ir: CRC error: rx=0x0000 calc=0x9313 len=11 ctrl=0x20 data_len=2  ← 回声帧损坏

基站端：
W (xxx) app_ir: CRC error: rx=0x0400 calc=0x1965 len=9 ctrl=0x20 data_len=0   ← 收到损坏数据
```

**失败原因分析**：
1. **红外回声问题**：设备端发送时会接收到自己的回声，导致数据混乱
2. **硬件限制**：红外接收头响应速度不够，无法正确处理连续发送
3. **数据损坏**：`data_len` 字段从 13 变成 0 或 208，说明传输过程中数据被严重损坏

**尝试的解决方案**：
1. 修复 CRC 计算范围 → 未解决
2. 降低波特率（4800 → 1200）→ 未解决

**结论**：红外双向通讯在当前硬件条件下不可靠，需要改用单向通讯方案。

## 11. 新方案：红外单向通讯

### 11.1 设计思路

**核心改变**：
- 取消基站端到设备端的红外请求
- 设备端持续通过红外发送自己的 CAN 数据
- 基站端持续接收所有设备端的红外数据

**优势**：
1. 避免双向通讯的回声问题
2. 简化协议，无需请求-应答机制
3. 设备端主动发送，实时性更好

### 11.2 通讯流程

```
设备端 #1 (ID=0xD0)                          基站端 (ID=0xA0)
    │                                            │
    │  持续接收功率板CAN报文                        │
    │  缓冲区: [CAN_ID=0x101, 最新数据]            │
    │                                            │
    │  每 50ms 发送一次                            │
    │── IR: [设备ID=0xD0] [CAN数据] ─────────────→│
    │                                            │  解析设备ID
    │                                            │  转发到发送端功率板
    │                                            │
    │  50ms 后继续发送                              │
    │── IR: [设备ID=0xD0] [CAN数据] ─────────────→│
    │                                            │


设备端 #2 (ID=0xD1)                          基站端 (ID=0xA0)
    │                                            │
    │  每 50ms 发送一次（与设备#1错开）              │
    │── IR: [设备ID=0xD1] [CAN数据] ─────────────→│
    │                                            │  解析设备ID
    │                                            │  转发到发送端功率板
```

### 11.3 帧格式

**简化帧格式**（单向）：
```
┌──────────┬──────────┬─────────────────────┬──────┬──────┐
│ 设备ID   │ CAN_DLC  │ CAN_DATA            │ seq  │ CRC  │
│ 1 byte   │ 1 byte   │ 0-8 bytes           │ 1B   │ 2B   │
└──────────┴──────────┴─────────────────────┴──────┴──────┘
```

**总长度**：1 + 1 + 8 + 1 + 2 = 13 字节（固定）

### 11.4 时序设计

```
时间轴: 0ms    12.5ms  25ms    37.5ms  50ms    62.5ms  75ms    87.5ms  100ms
         │       │       │       │       │       │       │       │       │
设备#0:  │ ──── TX ──── │       │       │ ──── TX ──── │       │       │
设备#1:  │       │       │ ──── TX ──── │       │       │ ──── TX ──── │
设备#2:  │       │       │       │       │       │       │       │       │
设备#3:  │       │       │       │       │       │       │       │       │

每个设备每 50ms 发送一次
设备之间错开 12.5ms，避免冲突
```

### 11.5 实现要点

**设备端**：
1. 持续接收功率板的 CAN 报文，缓存最新帧
2. 每 50ms 从缓冲区取最新帧
3. 构造简化帧（设备ID + CAN数据 + CRC）
4. 通过红外发送

**基站端**：
1. 持续接收红外数据
2. 解析帧，提取设备ID和CAN数据
3. 转发到发送端功率板
4. 更新设备在线状态

### 11.6 优势对比

| 特性 | 双向通讯（失败） | 单向通讯（新方案） |
|------|------------------|-------------------|
| 协议复杂度 | 高（请求-应答） | 低（单向发送） |
| 回声问题 | 严重 | 无 |
| 实时性 | 低（等待请求） | 高（主动发送） |
| 可靠性 | 低 | 高 |
| 帧长度 | 22 字节 | 13 字节 |
