## 1. 总体目标

### 1.1 工程目标

- 基于 **ESP32-C3 + ESP-IDF** 完成板级固件框架搭建。
- 支持以下核心功能：
  - CAN 通讯（TWAI）
  - 红外串口链路
  - ESP-NOW 设备组网
  - I2C 屏幕显示调试信息
  - WS2812B 状态灯驱动
  - UART0 串口收发
- 形成可长期维护的模块化工程结构，便于后续扩展协议、调试页面和设备管理逻辑。

### 1.2 业务目标

本工程是**无线充电系统的外置通讯板**，用于无线识别和信息交互。

**一发四收**无线充电系统：
- 1 个发射端（发送端功率板 + 基站端通讯板）
- 4 个接收端（接收端功率板 + 设备端通讯板）

**系统架构**：
- 基站端（ID: 0xA0）通过 CAN 连接发送端功率板
- 设备端（ID: 0xD0-0xD3）通过 CAN 连接接收端功率板
- 每个设备端配合一个接收端，形成独立的接收单元

**核心原则**：
1. **点对点通讯**：每次交互只与一个设备通讯
2. **红外识别**：通过红外发现设备并确认身份
3. **轮询调度**：基站按周期依次与每个设备通讯
4. **请求-应答**：基站发起请求，设备回复应答

**通讯方式**：
- **方式 1：红外透传 CAN 数据（基础功能）**
  - 基站持续接收发送端功率板的 CAN 报文，缓存最新一帧
  - 设备持续接收接收端功率板的 CAN 报文，缓存最新一帧
  - 基站通过红外轮询设备，发送最新 CAN 帧，请求设备回复
  - 设备收到请求后，立即回复最新 CAN 帧
  - 20Hz 轮询（50ms 周期），无握手，简单高效
- **方式 2：ESP-NOW 透传 CAN 数据（扩展功能）**
  - 先通过红外识别确认目标设备
  - 建立 ESP-NOW 点对点连接
  - 通过 ESP-NOW 高速透传 CAN 数据
  - 延迟 <5ms，可靠性 >99%

**详细设计**：见 `doc/wireless_charging_design.md`

### 本轮进度更新（红外双向通讯测试失败，2026-05-06）

- 测试内容：
  - 红外双向通讯：基站端发送 CMD 帧，设备端回复 RSP 帧
  - 测试波特率：4800、1200
- 测试结果：❌ 失败
- 失败现象：
  - 设备端能正确接收基站的 CMD 帧
  - 设备端发送 RSP 帧后，收到自己发送的回声帧，但数据被损坏
  - 基站端无法接收到设备端的 RSP 帧
- 失败原因：
  - 红外回声问题：设备端发送时会接收到自己的回声，导致数据混乱
  - 硬件限制：红外接收头响应速度不够，无法正确处理连续发送
  - 数据损坏：`data_len` 字段从 13 变成 0 或 208，说明传输过程中数据被严重损坏
- 尝试的解决方案：
  - 修复 CRC 计算范围 → 未解决
  - 降低波特率（4800 → 1200）→ 未解决
- 结论：红外双向通讯在当前硬件条件下不可靠，需要改用单向通讯方案

### 新方案：红外单向通讯

- 设计思路：
  - 取消基站端到设备端的红外请求
  - 设备端持续通过红外发送自己的 CAN 数据
  - 基站端持续接收所有设备端的红外数据
- 优势：
  - 避免双向通讯的回声问题
  - 简化协议，无需请求-应答机制
  - 设备端主动发送，实时性更好
- 简化帧格式：
  - [设备ID] [CAN_DLC] [CAN_DATA] [seq] [CRC]
  - 总长度：13 字节（固定）
- 时序设计：
  - 每个设备每 50ms 发送一次
  - 设备之间错开 12.5ms，避免冲突
- 详细设计：见 `doc/wireless_charging_design.md` 第 11 节
